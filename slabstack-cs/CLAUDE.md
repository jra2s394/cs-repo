# CLAUDE.md — Slabstack CS Operations

## Role & Context

This repo supports the Customer Success Operations function at **Slabstack**, a software platform for ready-mix and concrete producers. The work here is content and documentation, not software engineering.

**My role:** Customer Success Operations — building scalable resources that reduce time-to-value for customers and support the sales and CS teams.

---

## Tools & Integrations

- **Intercom** — Knowledge base articles live here; format all KB content for Intercom (clear headers, short paragraphs, scannable structure)
- **Asana** — Project and task tracking for CS initiatives
- **Shortcut** — Used for tracking content production tasks and sprint work

## What I'm Building

- **Onboarding videos** — 5-part self-serve series (2–3 min each), word-for-word scripts, covering core Slabstack workflows
- **Knowledge base articles** — 16-article plan derived from ~1,040 Intercom support conversations; formatted for search and self-resolution
- **Pre-sales docs** — Discovery frameworks, demo guides, objection handling, ROI calculators
- **Integration guides** — Technical onboarding docs for integrations (Sysdyne is first)

---

## Content Standards

- Write for concrete industry operators, not technical audiences — plain language, practical focus
- KB articles: short paragraphs, numbered steps for procedures, headers for scannability
- Video scripts: conversational tone, second person ("you"), clear action per section
- Integration guides: prerequisites first, step-by-step setup, troubleshooting section, FAQ
- No marketing fluff — assume the reader is already a customer trying to get something done

---

## Behavior Rules

- **Do not take irreversible actions without asking first.** This includes deleting files, publishing to Intercom, creating/closing Asana tasks or Shortcut tickets, or sending any external communication.
- Confirm before creating, modifying, or removing anything in external systems (Intercom, Asana, Shortcut).
- When drafting content, present it for review before treating it as final.
- Prefer editing existing files over creating new ones unless a new file is clearly needed.
